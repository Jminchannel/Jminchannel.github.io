Redis基础_缓存

什么是缓存

缓存是数据交换的缓冲区，称作Cache，是存贮数据的临时地方，通常读写性能较高。

读写性能：CPU 》内存 》磁盘

通常CPU运算需要先通过磁盘或者内存读取数据放到寄存器中，导致运算性能受限。为缓解限制，在CPU中出现了缓存，将经常读写的数据存储到缓存中。

缓存的作用：

  1. 降低后端负载
  2. 提高读写效率，降低响应实践

缓存的成本：

  1. 数据一致性成本
  2. 代码维护成本
  3. 运维成本

添加Redis缓存

设置redis缓存之前的操作：通常客户端向服务器发起请求后，服务器将请求打到数据库，从数据库获取数据并向客户端返回。

设置redis缓存之后：在客户端和数据库之间添加了中间层Redis，客户端向服务器发起请求后，服务器先访问redis，查询是否存在所需数据，如果存在则将数据直接返回到客户端，无需访问数据库。如果不存在即未命中redis，则访问数据库并获取数据，先将数据缓存到redis中，再将数据返回给客户端。

缓存更新策略

  1. 内存淘汰

    无需自己维护，利用Redis的内存淘汰机制，当内存不足时自动淘汰部分数据。下次查询时更新缓存。一致性较差，零维护。

  2. 超时剔除

    给缓存数据添加TTL时长，到期后自动删除缓存，下次查询时更新缓存。一致性一般，低维护。

  3. 主动更新

    编写业务逻辑，在修改数据库同时，更新缓存。高一致性，高维护。

业务场景

  • 低一致性需求：使用内存淘汰机制。例如店铺类型的查询缓存
  • 高一致性需求：主动更新，并以超时剔除作为兜底方案。例如店铺详情查询的缓存

主动更新策略

  1. Cache Aside Pattern

    即人工编码，由缓存调用者在更新数据库的同时更新缓存。

  2. Read/Write Through Pattern

    缓存与数据库整合为一个服务，由服务来维护一致性。调用者调用该服务，无需关心缓存一致性问题。

  3. Write Behind Caching Pattern

    调用者只操作缓存，由其它线程异步的将缓存数据持久化到数据库，保证最终一致。

通常采取第一种主动更新策略，人工编码。

操作缓存和数据库时有三个问题需要考虑：

  1. 删除缓存还是更新缓存？

    选择删除缓存。更新选择是当数据发生改变时，缓存随着更新，查询次数比较少的情况下，缓存操作频繁；删除缓存是当数据一旦发生变动，相应缓存删除，直到该数据被查询才得以插入新的缓存数据。

  2. 如何保证缓存与数据库的操作的同时成功或失败？

    • 在单体系统，将缓存与数据库操作放在一个事务。
    • 在分布式系统，利用TCC等分布式事务方案。
    • 还要考虑线程安全！

  3. 先操作缓存还是先操作数据库？

    • 先删除缓存，再操作数据库

    • 先操作数据库，再删除缓存，首选

      尽可能避免异常情况，可以在写入缓存的时候加入缓存有效期

缓存的最佳实践方案

  1. 低一致性需求：使用Redis自带的内存淘汰机制
  2. 高一致性需求：主动更新，并以超时剔除作为兜底方案

读操作：

  • 缓存命中直接返回
  • 缓存未命中，查询数据库，并写入缓存，设定超时时间

写操作：

  • 先写数据库，然后再删除缓存
  • 要确保数据库与缓存操作的原子性

缓存穿透

缓存穿透，客户端请求的数据在缓存和数据库中均不存在，缓存永远不会生效，这些请求都会打到数据库。

常见的解决方案有两种：

  1. 缓存空对象

    将null缓存，简单粗暴，方便维护；但造成额外的内存消耗，短期的不一致。

  2. 布隆过滤

    优势，内存占用较少，没有多余的key

    缺陷，实现复杂，存在误判可能
    通常采取方案一，缓存空对象

    还有解决方案：

    1. 增强id的复杂度，避免id被猜测
    2. 做好数据的基础格式校验
    3. 加强用户权限校验
    4. 做好热点参数的限流

缓存雪崩

缓存雪崩，同一时段大量的缓存key同时失效或者Redis服务器宕机，达到大量请求到达数据库，带来巨大压力。

解决方案

  1. 给不同的Key的TTL添加随机值
  2. 利用Redis集群提高服务的可用性
  3. 给缓存业务添加降级限流策略（让请求快速失败，牺牲服务）
  4. 给业务添加多级缓存

缓存击穿

缓存击穿问题也叫热点Key问题，一个被高并发（同时）访问并且缓存重建业务较复杂的key突然失效了，无数的请求访问会在瞬间给数据库带来巨大的冲击。

常见的解决方案

  1. 互斥锁

    优势：编码简单，劣势：线程需要等待，而且存在死锁分享，性能较差

    互斥锁可以用redis中的setnx实现（当key不存在时可以创建键值对，如果存在无法修改该键值对，只能删除重建）

    java对应的方法为

    stringRedisTemplate.opsForValue.setIfAbsent(key,value)
